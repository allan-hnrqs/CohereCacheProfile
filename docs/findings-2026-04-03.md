### Findings: 2026-04-03

This note summarizes the first live Chat API profiling run saved in [results/chat-cache-2026-04-03.json](../results/chat-cache-2026-04-03.json).

If you want the broader follow-up that covers prompt sizes, natural-language prefixes, and multi-turn `messages` history, read [docs/chat-shapes-2026-04-03.md](chat-shapes-2026-04-03.md) first.

#### Scope

- endpoint: `POST /v2/chat`
- models:
  - `command-a-03-2025`
  - `command-a-reasoning-08-2025`
  - `command-r7b-12-2024`
- smoke checks:
  - `POST /v2/embed`
  - `POST /v2/rerank`
- prompt shapes:
  - tiny prompt
  - tiny prompt with a short system message
  - large repetitive system prefix
  - large unique system prefix
  - exact repeats
  - near-miss variants with an early prefix mutation

#### Measured Facts

##### 1. `usage.cached_tokens` is present in Cohere's schema, but not documented as a public feature surface

The public API schema on the Chat reference includes `usage.cached_tokens` with the description:

`The number of prompt tokens that hit the inference cache.`

What is missing from public docs:

- no request parameter to enable or control caching
- no pricing section for cache hits
- no lifecycle, TTL, or correctness guarantees
- no explanation of model support or endpoint support

Live smoke tests complicate this further:

- `v2/chat` returned `usage.cached_tokens`
- `v2/embed` did not return `cached_tokens` in `meta`
- `v2/rerank` did not return `cached_tokens` in `meta`

So even if the schema is shared internally, the field is not consistently present in actual runtime responses.

##### 2. Command A models showed no usable prompt-cache signal

For both:

- `command-a-03-2025`
- `command-a-reasoning-08-2025`

Results were stable:

- `cached_tokens` stayed `0`
- repeated exact large prefixes did not reduce billed input
- near-miss variants also behaved the same in billing

Examples:

- `command-a-03-2025`, repetitive large exact:
  - billed input: `16817`
  - raw input: `17346`
  - cached tokens: `0`
- `command-a-03-2025`, unique large exact:
  - billed input: `35013`
  - raw input: `35542`
  - cached tokens: `0`

##### 3. Command A Reasoning adds a much larger hidden prompt footprint

Tiny-prompt runs showed:

- `command-a-03-2025`: `raw_input_tokens - billed_input_tokens = 495`
- `command-r7b-12-2024`: `raw_input_tokens - billed_input_tokens = 495`
- `command-a-reasoning-08-2025`: `raw_input_tokens - billed_input_tokens = 1386`

This suggests the public `tokens.input_tokens` metric includes hidden model-side scaffolding that is not fully reflected in `billed_units.input_tokens`.

That is not necessarily a bug, but it should be explicitly documented because it affects how developers interpret telemetry.

##### 4. `command-r7b-12-2024` reports non-zero `cached_tokens`, but the behavior does not match normal cross-request prompt caching

Observed on tiny prompts:

- tiny plain prompt:
  - billed input: `3`
  - raw input: `498`
  - cached tokens: `496`
- tiny prompt with system message:
  - billed input: `8`
  - raw input: `537`
  - cached tokens: `528`

That means `cached_tokens` can be non-zero on what is effectively the first observed request. That already makes it unlike standard public prompt caching.

##### 5. On `command-r7b-12-2024`, highly repetitive prompts produce extreme cache reports

For the repetitive large prefix:

- `large_exact_1`:
  - billed input: `16817`
  - raw input: `17346`
  - cached tokens: `17344`
- `large_exact_2`:
  - billed input: `16817`
  - raw input: `17346`
  - cached tokens: `17344`

One near-miss variant also showed:

- `large_miss_2`:
  - billed input: `16822`
  - raw input: `17351`
  - cached tokens: `17344`

But another near-miss did not:

- `large_miss_1`:
  - billed input: `16822`
  - raw input: `17351`
  - cached tokens: `512`

This pattern is not compatible with a clean public prefix-cache contract.

##### 6. On `command-r7b-12-2024`, unique large prefixes do not show cache growth across repeated exact requests

For the non-repetitive large prefix:

- `unique_large_exact_1`:
  - billed input: `35013`
  - raw input: `35542`
  - cached tokens: `512`
- `unique_large_exact_2`:
  - billed input: `35013`
  - raw input: `35542`
  - cached tokens: `512`

Near-miss variants were similar:

- `unique_large_miss_1`: `528`
- `unique_large_miss_2`: `512`

This is the strongest evidence against user-visible cross-request caching on `r7b` for normal conversation history prefixes.

##### 7. Non-chat endpoint smoke tests did not expose cache telemetry

Saved in [results/nonchat-cache-smoke-2026-04-03.json](../results/nonchat-cache-smoke-2026-04-03.json).

Observed:

- `v2/embed` with `embed-v4.0`
  - exact runs billed `4800` input tokens
  - near-miss billed `4804` input tokens
  - no `cached_tokens` field returned
- `v2/rerank` with `rerank-v3.5`
  - exact and near-miss runs all billed `1` search unit
  - no `cached_tokens` field returned

These were only smoke tests, not deep endpoint studies. Still, they show that cache telemetry is not uniformly surfaced across endpoints.

#### Working Interpretation

The simplest explanation consistent with the current data is:

- `cached_tokens` exists as backend telemetry for inference-cache hits
- the reported value is model-dependent
- some of it likely reflects hidden scaffold reuse, not user-managed prompt-prefix reuse
- repetitive prompts on `r7b` may trigger a separate optimization path or inconsistent accounting

That is an inference from the measurements, not a documented Cohere guarantee.

#### Suspected Docs Gaps or Bugs

These deserve follow-up with Cohere support or docs:

1. `usage.cached_tokens` is present in the schema but effectively undocumented in the public guides.
2. The relation between `tokens.input_tokens`, `billed_units.input_tokens`, and `cached_tokens` is not explained.
3. `command-r7b-12-2024` reports very large cache hits on repetitive prompts, including on first-observed requests.
4. Repeated exact requests do not reduce billed input even when `cached_tokens` is large.
5. Non-chat endpoints do not consistently return the same cache telemetry that the shared schema appears to advertise.

#### What We Can Say Safely Right Now

- There is no evidence here of Anthropic-style public prompt caching on Cohere Chat.
- There is evidence that Cohere tracks some notion of inference-cache hits internally.
- That signal is not consistent enough to base a product architecture or cost model on.
